# pnpm 怎样管理依赖

pnpm 的名字来自 Performant npm。它通过内容寻址存储和链接复用包文件，减少多个项目重复占用磁盘，同时让依赖关系更接近 `package.json` 中的声明。

## 1. 内容寻址存储

pnpm 会把包文件保存到全局 store。文件内容相同，就可以复用同一份存储结果。项目里的 `node_modules/.pnpm` 再通过硬链接或复制优化机制引用这些文件。

如果一百个项目使用相同版本的 `lodash`，磁盘不需要保存一百份完全相同的内容。实际节省量取决于文件系统、pnpm 配置和各项目使用的版本。

## 2. 符号链接和依赖结构

pnpm 用符号链接把项目直接依赖暴露到根 `node_modules`。每个包能访问的依赖则按照它自己的声明组织。

这种结构可以减少幽灵依赖。项目代码如果导入一个没有写进 `package.json` 的包，通常会更早暴露错误。某些依赖了扁平 `node_modules` 假设的旧工具可能需要额外配置。

## 3. 使用 pnpm 的理由

- 多项目共用 store，磁盘占用通常更低。
- 已缓存的内容可以直接复用，安装速度较快。
- 对未声明依赖的限制更严格。
- 内置 workspace 适合管理 Monorepo。

这些优势不意味着每个项目都要迁移。团队已有稳定的 npm 或 Yarn 流程时，还要计算 lock 文件变化、缓存配置、部署平台和成员学习成本。

## 4. 适用场景

经常创建项目、维护 Monorepo，或磁盘被多个 `node_modules` 占满时，pnpm 很值得尝试。迁移后应在持续集成环境重新安装并运行完整测试，不能只检查本地开发服务器是否启动。

包管理器最终要解决的是可复现安装。速度和磁盘占用很重要，lock 文件、依赖声明和供应链安全同样重要。
