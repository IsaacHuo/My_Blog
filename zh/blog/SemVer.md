# 语义化版本 SemVer 解释

项目依赖一多，版本号就不能随便增长。语义化版本（Semantic Versioning，SemVer）用一套公开规则说明每次发布改了什么，以及升级是否可能破坏旧代码。

## 1. 版本号格式

SemVer 使用 `MAJOR.MINOR.PATCH`，也就是主版本号、次版本号和修订号。例如 `1.4.2`。

1. `MAJOR`：出现不兼容的 API 修改时增加。删除函数或改变参数定义，都可能让旧代码无法运行。
2. `MINOR`：增加向下兼容的功能时增加。例如新增方法，或给现有方法加入可选参数。
3. `PATCH`：修复向下兼容的问题时增加，例如修复 Bug，同时保持原有 API。

版本号后面还可以附加先行版本标识，例如 `1.0.0-alpha.1`。这类版本通常用于测试，稳定性和兼容承诺可能低于正式版本。

## 2. npm 中的版本范围

`package.json` 可以写死一个版本，也可以声明允许安装的范围。

### `^`，Caret

`^1.2.3` 表示 `>=1.2.3 <2.0.0`。它允许次版本号和修订号更新，只要最左侧的非零版本号不变。npm 保存依赖时通常会使用这种写法。

零开头的版本要单独留意。`^0.2.3` 的范围小于 `0.3.0`，因为 SemVer 将 `0.x` 视为仍在初始开发阶段。

### `~`，Tilde

`~1.2.3` 表示 `>=1.2.3 <1.3.0`，通常只接受同一次版本下的修订更新。希望获得 Bug 修复，同时减少功能变化时，可以使用它。

### 固定版本

`1.2.3` 只接受这个版本。它限制最严格，但仍需配合 lock 文件固定整棵依赖树，因为直接依赖的子依赖也有自己的版本范围。

## 3. 版本范围和 lock 文件各管什么

版本范围描述项目愿意接受哪些版本，`package-lock.json`、`pnpm-lock.yaml` 或 `yarn.lock` 记录一次解析后的确切结果。

允许范围更新可以获得新功能和修复，也会把上游发布失误带进项目。如果包作者在 PATCH 版本里修改了 API，即使你的版本声明看起来安全，构建仍可能失败。

因此，应用项目通常应提交 lock 文件，并在持续集成环境中使用确定性安装命令。库项目则需要认真声明依赖范围，让使用者有合理的升级空间。
