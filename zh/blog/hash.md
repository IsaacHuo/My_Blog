# 什么是哈希

哈希（Hash），也叫散列或摘要，是把任意长度的输入映射为固定长度输出的函数。常见的哈希值会写成十六进制字符串。

哈希和加密解决的问题不同。加密需要持有密钥的一方恢复原文，密码学哈希通常设计成难以逆向，只用于比较、验证或建立索引。

## 1. 密码学哈希的常见特性

1. 确定性：相同输入产生相同输出。
2. 原像抗性：给定哈希值，很难找到能产生它的原始输入。
3. 雪崩效应：输入只变化一个 bit，输出也会发生大幅变化。
4. 碰撞抗性：很难找到两个不同输入，使它们得到相同输出。

碰撞在数学上一定存在，因为输入空间无限，固定长度的输出空间有限。安全性要求是让攻击者在可接受的时间和算力内找不到碰撞。

普通哈希表使用的哈希函数更关注速度和分布均匀，未必具有这些密码学性质。

## 2. 常见算法

- MD5 输出 128 bit，已经不能用于抵抗恶意碰撞。它仍会出现在非安全场景的文件校验中。
- SHA-1 输出 160 bit，也已出现实际碰撞，不应继续用于数字签名等安全用途。
- SHA-2 包含 SHA-256、SHA-512 等算法，目前仍广泛用于完整性校验和密码学协议。
- bcrypt、scrypt 和 Argon2 用于密码派生。它们故意消耗更多时间或内存，以提高暴力破解成本。

## 3. npm 为什么需要哈希

安装 `react@18.0.0` 之类的包时，Registry 元数据会提供完整性信息。npm 下载压缩包后重新计算哈希，再和元数据中的值比较。

两者相同，说明收到的内容与发布记录一致。两者不同，npm 会认为文件损坏或遭到修改，并停止安装。

`package-lock.json` 中的 `sha512-...` 字段保存了这类完整性信息。它能发现内容变化，但安全边界仍取决于 Registry、发布账号和构建来源是否可信。

## 4. 其他应用

- 密码存储：系统应保存经过盐值和专用密码哈希处理的结果，不能直接保存明文，也不该只做一次普通 SHA-256。
- Git：提交和对象通过哈希标识。SHA-1 曾是默认算法，新版本 Git 也支持 SHA-256 仓库格式。
- 哈希表：利用哈希函数把键映射到桶，平均情况下可以做到接近 `O(1)` 的查询。

把哈希理解成数据指纹很方便，但指纹这个比喻有边界。不同数据仍可能发生碰撞，具体安全程度取决于算法和使用方式。
